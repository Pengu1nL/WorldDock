# 界坞 WorldDock 前端设计需求文档

文档版本：v0.2
更新日期：2026-05-25
适用阶段：前端体验原型 / 高保真 Mock 阶段
关联文档：`docs/jiewu_worlddock_prd.pdf`

## 1. 文档目标

本文档用于定义界坞 WorldDock 在前端先行阶段的 UI / UX 需求。

当前阶段的目标不是完成后端能力，也不是制定具体开发排期，而是先做出一个足够完整、可信、可交互的前端产品雏形，让用户能够体验到：

- 从一句灵感创建世界；
- 与 Agent 协作推演世界；
- 将 Agent 输出沉淀为世界档案；
- 从世界冲突中孵化故事种子；
- 浏览、发布、Star、Fork 类 GitHub 的世界仓库；
- 感知界坞 Local 与界坞 Cloud 的体验差异。

前端可以使用 Mock 数据和 Mock Agent，但必须呈现真实产品体验的结构、状态、反馈和关键决策点。

## 2. 设计范围

### 2.1 本阶段包含

- 产品整体信息架构；
- 核心页面 UI / UX 需求；
- 核心用户路径；
- 对话优先的工作台体验；
- Agent 流式输出、建议沉淀与辅助信息收起体验；
- 世界档案、故事种子、冲突、一致性提醒等核心对象的前端表现；
- 界仓社区公开世界仓库体验；
- Local / Cloud 状态差异的前端表达；
- Mock 数据与 Mock Agent 行为要求；
- 视觉风格、布局、交互状态、响应式要求；
- 前端验收标准。

### 2.2 本阶段不包含

- 后端真实实现；
- 数据库设计落地；
- 真实模型调用；
- 真实计费结算；
- 真实 Push / Publish 后端链路；
- 真实内容审核系统；
- 具体技术选型与工程实施排期；
- 最终品牌视觉规范。

## 3. 产品体验定位

界坞前端体验应被设计为“世界观创作者的专业工作台”，而不是普通 AI 聊天页面、小说生成器或营销型网站。

核心体验心智：

- 本地是 World IDE；
- 网站是 World GitHub；
- 工作台是对话推演现场；
- 世界档案是长期资产；
- 故事种子是世界内部矛盾自然生长出的叙事入口；
- Agent 是协作者和审校者，不是替用户自动写完作品的工具。

前端首屏与核心流程必须持续强化以下判断：

用户不是来获得一篇小说，而是来创造一个会继续长出故事的世界。

## 4. 目标用户与体验诉求

### 4.1 普通创作者

诉求：

- 打开网站即可开始；
- 不需要理解模型配置；
- 希望 Agent 能提出有启发的问题；
- 希望灵感能被整理为结构化设定；
- 希望故事种子足够有画面感和继续发展的欲望。

前端重点：

- 创建流程轻；
- 语言友好但不幼稚；
- Agent 输出要可读、可保存、可修改；
- 操作路径清晰，避免技术感过重。

### 4.2 折腾型创作者 / 初级开发者

诉求：

- 可以本地部署；
- 能配置自己的模型 Key；
- 能看到模型连接状态；
- 能明确知道哪些数据会 Push 到社区；
- 能掌控公开与私密边界。

前端重点：

- Local 状态清晰；
- 模型连接测试反馈明确；
- Push 前预览范围和差异；
- 隐私边界必须可视化。

### 4.3 社区浏览者

诉求：

- 快速发现有趣世界；
- 快速理解公开世界的亮点；
- 查看设定、冲突和故事种子；
- 决定是否 Star 或 Fork。

前端重点：

- 世界仓库页要有 GitHub-like 的仓库心智；
- Overview 能快速说明世界吸引力；
- Fork 来源、授权和版本记录要清晰。

### 4.4 平台管理员

诉求：

- 查看用户、用量、公开世界和举报；
- 控制模型成本；
- 下架违规内容；
- 暂停模型调用。

前端重点：

- 管理后台以信息密度和操作安全为主；
- P0 可以是基础可用界面，不追求复杂数据分析。

## 5. 前端设计原则

### 5.1 推演优先

前端主按钮和主流程不应围绕“写小说”“生成正文”“一键成书”设计。

推荐表达：

- 继续推演；
- 扩展设定；
- 检查一致性；
- 收束为设定；
- 生成故事种子；
- 保存到档案。

避免表达：

- 生成小说；
- 一键写书；
- 写第一章；
- 自动续写全文。

### 5.2 对话优先

与 Agent 对话进行世界推演，是界坞工作台的第一主体验。

工作台默认状态不应把多个信息栏同时摊开，避免用户一进入页面就被资产树、建议面板、上下文列表和状态信息包围。

默认界面应优先呈现：

- 清晰、宽敞、可长时间阅读的对话流；
- 明确的 Agent 模式切换；
- 易输入、易追问、易停止、易重试的输入区；
- 当前世界状态的轻量提示；
- 本轮可沉淀内容的轻量入口。

避免：

- 默认三栏同时展开；
- 让右侧建议面板持续挤压对话宽度；
- 在对话区旁边长期堆叠过多状态卡片；
- 把工作台做成信息仪表盘，而不是推演现场。

### 5.3 沉淀可触达

用户必须理解对话如何转化为世界资产，但这种理解不必依赖常驻三栏。

沉淀信息应通过以下方式触达：

- Agent 消息内出现“可保存设定”“潜在冲突”“故事种子”等内联建议；
- 输入框上方或对话右上角出现轻量待处理计数；
- 点击建议后打开抽屉、弹层或详情页；
- 保存后以 Toast、数量变化、资产入口高亮等方式反馈；
- 在档案、冲突池、故事种子等独立页面集中管理。

工作台需要让沉淀动作足够顺手，但不能让沉淀面板抢走对话主舞台。

### 5.4 用户确认

Agent 不能在前端体验中表现为“自动修改世界”。

所有结构化写入都必须经过用户操作：

- 保存；
- 修改后保存；
- 丢弃；
- 暂存；
- 标记为待定；
- 标记为废弃。

### 5.5 上下文可解释

Agent 的结果不能像黑箱。

前端需要能解释本轮使用了哪些上下文，但默认不应常驻展示完整上下文列表。

可通过以下方式呈现：

- Agent 消息底部的“本轮引用 4 项上下文”入口；
- 对话顶部的上下文选择器；
- 右侧上下文抽屉；
- 进入独立档案页后查看完整关联关系。

上下文示例：

- 世界规则：记忆交易；
- 势力：记忆银行；
- 冲突：记忆是财产还是人格；
- 最近变更：新增“情感剥离限制”。

用户应能理解 Agent 为什么这样回答，并能在后续版本中手动增删上下文。

### 5.6 状态比装饰重要

界坞是创作工具，UI 应优先服务于信息组织、状态反馈和长时间使用。

需要突出：

- 世界成熟度；
- 发布状态；
- 本地 / 云端模式；
- 模型连接状态；
- 本次消耗；
- 余额；
- 是否有未保存建议；
- 是否有未发布更改；
- 是否存在一致性风险。

### 5.7 Mock 也要像真实产品

Mock 不应只是静态假数据。

前端原型需要模拟：

- Agent 流式输出；
- 工具调用状态；
- 归档建议逐步出现；
- usage 更新；
- 保存后档案数据变化；
- Fork 后生成私有世界；
- Push / Publish 前后的状态变化；
- 错误、空状态、余额不足、模型不可用等异常状态。

## 6. 整体信息架构

### 6.1 网站公开区

- `/`：产品入口，可轻量介绍产品并引导进入 Explore 或创建世界；
- `/explore`：发现世界；
- `/trending`：热门世界；
- `/topics/:tag`：标签世界列表；
- `/:username`：用户主页；
- `/:username/:worldSlug`：公开世界仓库首页；
- `/:username/:worldSlug/archive`：公开世界档案；
- `/:username/:worldSlug/seeds`：公开故事种子；
- `/:username/:worldSlug/conflicts`：公开冲突池；
- `/:username/:worldSlug/forks`：Fork 列表；
- `/:username/:worldSlug/releases`：版本记录。

### 6.2 网站应用区

- `/app/worlds`：我的世界；
- `/app/worlds/new`：创建世界；
- `/app/worlds/:worldId/workbench`：世界工作台；
- `/app/worlds/:worldId/archive`：世界档案；
- `/app/worlds/:worldId/seeds`：故事种子池；
- `/app/worlds/:worldId/conflicts`：冲突池；
- `/app/worlds/:worldId/publish`：发布设置；
- `/app/worlds/:worldId/settings`：世界设置。

### 6.3 设置区

- `/settings/profile`：个人资料；
- `/settings/billing`：用量与余额；
- `/settings/tokens`：Access Token 管理；
- `/settings/data`：导入导出；
- `/settings/security`：账号安全；
- `/settings/model`：本地模型配置，仅 Local；
- `/settings/community`：本地社区连接，仅 Local；
- `/settings/system`：系统状态与日志，仅 Local。

### 6.4 管理后台

- `/admin`：后台首页；
- `/admin/users`：用户管理；
- `/admin/worlds`：公开世界管理；
- `/admin/reports`：举报处理；
- `/admin/usage`：用量；
- `/admin/costs`：成本；
- `/admin/models`：模型配置。

管理后台在前端原型中可做基础体验，不作为第一视觉重点。

## 7. 视觉风格要求

### 7.1 总体气质

界坞应呈现克制、专业、沉浸、可信的创作工具感。

关键词：

- 结构清晰；
- 信息密度适中偏高；
- 状态明确；
- 安静但有想象力；
- 长时间使用不疲劳；
- 有世界观创作氛围，但不喧宾夺主。

不应呈现：

- 夸张营销页；
- 过度装饰的 AI 工具；
- 纯聊天软件；
- 游戏官网；
- 小说阅读站；
- 一屏一个大 Hero 的展示型网站。

### 7.2 色彩

推荐方向：

- 基础背景使用冷静中性色；
- 内容区使用明确层级；
- 重点操作可使用低饱和蓝绿、青色或琥珀作为功能色；
- 警告、冲突、一致性风险使用明确但不过度刺眼的提示色；
- Local / Cloud 状态可通过小型状态标识区分。

要求：

- 不使用单一色相铺满整个产品；
- 不依赖大面积渐变表达高级感；
- 不使用装饰性光球、模糊色块作为主要背景；
- 公开世界页可以比工作台更有内容展示感，但仍需保持仓库心智。

### 7.3 字体与排版

要求：

- 中文阅读舒适；
- 工作台内标题克制，避免大字号压缩操作空间；
- 卡片标题、标签、状态、正文层级清晰；
- 长文本输出要有良好行高和段落间距；
- 表格和列表适合快速扫描；
- 按钮文本不应换行溢出；
- 移动端需要保证最小可读字号。

### 7.4 布局

工作台以桌面端为主要体验目标。

全产品布局原则：

- 核心创作页优先采用单主区布局；
- 管理类页面优先采用列表 + 抽屉、列表 + 详情页或分步流程；
- 不把三栏作为默认页面模板；
- 只有后台、发布差异预览等确实需要高密度对照的页面，才允许使用多列布局；
- 多列布局必须可折叠，且不能压缩主要阅读或编辑区域。

桌面端默认形态：

- 顶部状态栏；
- 中央对话推演区；
- 左侧仅保留窄导航或可折叠资产入口；
- 右侧默认不展开，仅在需要时打开上下文、归档建议、一致性提醒或详情抽屉；
- 档案、故事种子、冲突池、发布等复杂管理任务跳转到独立页面完成。

工作台不应默认采用完整三栏式布局。

桌面端可选辅助层：

- 左侧资产入口：用于快速跳转，不承载完整档案管理；
- 右侧辅助抽屉：用于查看本轮上下文、归档建议、风险提醒；
- 顶部世界切换 / 状态区：用于显示当前世界、模式、发布状态和用量；
- 命令面板或快捷菜单：用于快速打开档案、种子、冲突、发布页面。

布局优先级：

1. 对话阅读与输入空间；
2. 本轮推演状态与 Agent 控制；
3. 可保存建议的发现与处理；
4. 资产浏览与结构化管理；
5. 发布、设置、用量等低频任务。

移动端要求：

- 保持单列对话为核心；
- 使用底部导航或顶部菜单进入“档案 / 种子 / 冲突 / 设置”；
- 建议、上下文、保存表单使用底部抽屉；
- 核心流程可完成；
- 长列表、长输出、卡片编辑不能出现遮挡或不可操作。

## 8. 核心页面需求

### 8.1 我的世界列表页

目标：让用户快速进入已有世界、创建新世界、理解世界状态。

页面应包含：

- 世界搜索；
- 创建世界入口；
- 世界卡片列表；
- 类型和风格标签；
- 最近更新时间；
- 档案数量；
- 故事种子数量；
- 发布状态；
- Local / Cloud 状态。

世界卡片状态：

- 私有；
- 未公开；
- 已公开；
- 有未发布更改；
- 本地未连接社区；
- 本地有未 Push 更改；
- 远程版本已更新。

空状态要求：

- 引导用户输入一个灵感创建第一个世界；
- 不使用大段说明文字；
- 给出 2 到 3 个示例灵感。

### 8.2 创建世界页

目标：用轻量输入启动创作，而不是让用户填写复杂表单。

输入字段：

- 初始灵感，必填；
- 世界名称，可选；
- 类型，可选；
- 风格关键词，可选；
- 不想要的方向，可选。

交互流程：

1. 用户输入灵感；
2. 点击开始推演；
3. Mock Agent 生成世界雏形确认卡；
4. 用户确认、修改或重新生成；
5. 创建世界；
6. 进入世界工作台。

世界雏形确认卡包含：

- 世界标题建议；
- 一句话核心设定；
- 初始核心矛盾；
- 可推演方向；
- 风格判断；
- 第一轮追问。

体验要求：

- 用户在确认前要感到“Agent 已经理解我的灵感”；
- 生成结果不应显得像最终结论，而应是可继续推演的世界雏形；
- 允许用户编辑关键字段后再创建。

### 8.3 世界工作台

目标：让用户在一个简洁、宽敞、低干扰的界面中完成与 Agent 的世界推演，并能把有价值内容顺手沉淀为世界资产。

工作台的核心不是同时展示所有信息，而是让用户始终知道：

- 当前正在推演什么；
- Agent 为什么这样推演；
- 哪些内容值得保存；
- 保存后可以去哪里继续管理。

工作台默认界面应接近 GPT 式对话体验，但必须保留界坞的世界资产沉淀能力。

#### 8.3.1 顶部状态栏

显示：

- 世界名称；
- 世界成熟度；
- 当前模式：Local / Cloud；
- 发布状态；
- 本地模型连接状态，Local；
- 余额 / 本次预计消耗，Cloud；
- 导出入口；
- 发布或 Push 入口；
- 打开界仓社区入口。

状态栏应紧凑，不应占用过多垂直空间。

顶部状态栏不应变成复杂工具栏。

高频操作保留在工作台：

- 切换 Agent 模式；
- 停止生成；
- 重新生成；
- 打开上下文；
- 打开待处理建议；
- 进入发布 / Push。

低频操作跳转到对应页面：

- 完整档案管理；
- 故事种子池管理；
- 冲突池管理；
- 世界设置；
- 用量和模型设置。

#### 8.3.2 中央 Agent 对话推演区

这是工作台的主视觉和主交互区域。

包含：

- 对话历史；
- Agent 模式按钮；
- 用户输入框；
- 流式输出；
- 工具调用状态；
- 本次 token / 创作点消耗；
- 停止生成；
- 重新生成；
- 继续追问；
- 本轮上下文入口；
- 本轮待处理建议入口。

Agent 模式：

- 追问；
- 扩展；
- 挑刺；
- 推演后果；
- 收束设定；
- 生成种子；
- 检查一致性。

对话区体验要求：

- 对话列宽应适合长文本阅读，不宜过窄；
- 用户输入框应始终容易找到；
- Agent 输出应有明确结构，但不能像表格机器；
- 每种模式应有不同的输出风格；
- Agent 消息内可以嵌入轻量建议卡，但不能把一条消息切碎成大量卡片；
- 用户能看出 Agent 在“推演世界”，而不是单纯聊天；
- 页面滚动应以对话流为中心，不因侧边栏内容变化产生明显跳动。

#### 8.3.3 内联沉淀建议

Agent 输出后，应在对话消息内部或消息尾部出现轻量可操作建议。

建议类型：

- 可保存设定；
- 潜在冲突；
- 故事种子；
- 一致性风险；
- 待定问题；
- 需要用户确认的上下文。

建议呈现要求：

- 默认展示摘要，不展开长正文；
- 每条建议有明确操作：保存、编辑后保存、暂存、丢弃、继续推演；
- 多条建议可折叠为“发现 3 条可保存设定”；
- 点击编辑或查看详情时打开抽屉或弹层；
- 保存成功后在对话内更新状态，并同步资产数量；
- 不要求右侧常驻沉淀面板。

#### 8.3.4 可收起资产入口

包含：

- 世界总览；
- 世界规则；
- 势力；
- 角色；
- 地点；
- 历史事件；
- 冲突池；
- 待定设定；
- 故事种子；
- 最近对话。

每个分类应显示数量或状态提示。

资产入口不是完整管理区。

要求：

- 桌面端可以是左侧窄栏、顶部下拉、命令面板或折叠菜单；
- 默认不占据大面积宽度；
- 点击分类后优先跳转到独立页面；
- 也可以在工作台内打开轻量预览抽屉；
- 选中某个资产继续推演时，对话区应能感知当前讨论对象。

#### 8.3.5 辅助信息抽屉

辅助信息不默认常驻展示，而是按需展开。

抽屉内容可包括：

- 归档建议；
- 一致性提醒；
- 故事种子建议；
- 当前上下文；
- 关联世界资产；
- 本轮工具调用详情。

抽屉使用规则：

- 用户主动点击时展开；
- Agent 流式生成期间可用小状态提示告知“正在识别可归档内容”，但不强制展开；
- 展开后不能遮挡输入框关键操作；
- 关闭后对话区应恢复主要空间；
- 长内容应提供跳转到独立页面的入口。

归档建议详情包含：

- 类型；
- 标题；
- 摘要；
- 正文；
- 建议保存位置；
- 关联设定；
- 操作：保存、修改后保存、丢弃。

一致性提醒详情包含：

- 风险描述；
- 关联旧设定；
- 冲突原因；
- 建议修正方向；
- 操作：采纳修正、暂存、忽略。

故事种子建议详情包含：

- 标题；
- 一句话钩子；
- 触发事件；
- 核心冲突；
- 潜在主角；
- 未解问题；
- 操作：保存到种子池、继续推演、丢弃。

当前上下文详情包含：

- 本轮引用的规则；
- 本轮引用的角色 / 势力；
- 本轮引用的冲突；
- 最近变更；
- 用户显式选择的上下文。

### 8.4 世界档案页

目标：维护用户确认后的结构化世界设定。

页面结构：

- 顶部或左侧轻量分类筛选；
- 主体为设定卡片列表或表格列表；
- 详情使用抽屉、弹层或独立详情页；
- 编辑状态可进入独立编辑视图。

档案页不应默认采用“分类 + 列表 + 详情”三栏同时展开。

桌面端允许：

- 左侧轻量分类 + 主列表的双栏结构；
- 主列表 + 右侧详情抽屉，但抽屉默认关闭；
- 批量管理时使用表格视图。

移动端应使用：

- 顶部分类筛选；
- 单列列表；
- 点击卡片进入详情页或底部抽屉。

档案分类：

- 世界总览；
- 世界规则；
- 势力；
- 角色；
- 地点；
- 历史事件；
- 资源；
- 禁忌；
- 冲突；
- 待定设定；
- 废弃设定。

设定卡字段：

- 标题；
- 类型；
- 摘要；
- 正文；
- 状态；
- 关联设定；
- 关联故事种子；
- 创建来源；
- 更新时间。

操作：

- 新建设定；
- 编辑设定；
- 标记待定；
- 标记废弃；
- 查看关联内容；
- 从设定继续推演。

### 8.5 故事种子池

目标：集中管理从世界矛盾中生长出的故事入口。

故事种子卡字段：

- 标题；
- 一句话钩子；
- 触发事件；
- 核心冲突；
- 涉及设定；
- 潜在主角；
- 故事方向；
- 未解问题；
- 状态；
- 来源冲突。

操作：

- 继续推演；
- 查看关联设定；
- 生成变体；
- 提升冲突强度；
- 保存为高潜力；
- 废弃。

限制：

- 不做章节管理；
- 不做正文编辑器；
- 不提供“一键写第一章”作为主操作。

### 8.6 冲突池

目标：展示世界内部矛盾，并作为故事种子的上游。

冲突卡字段：

- 冲突标题；
- 冲突双方；
- 涉及规则；
- 涉及势力；
- 可能爆发事件；
- 可衍生故事种子；
- 状态。

操作：

- 从冲突生成故事种子；
- 推演后果；
- 查看相关设定；
- 标记为核心冲突；
- 标记为暂存。

### 8.7 公开世界仓库页

目标：让访问者快速理解一个公开世界，并决定是否 Star、Fork 或继续阅读。

页面结构：

- 顶部仓库信息；
- 导航：Overview / Archive / Seeds / Conflicts / Releases / Forks；
- 主体 README；
- 右侧元信息。

顶部信息：

- 世界名称；
- 作者；
- Star 数；
- Fork 数；
- 授权；
- 更新时间；
- Fork 来源，若有；
- 发布版本。

Overview 内容：

- 一句话设定；
- 世界类型；
- 风格标签；
- 核心规则；
- 核心矛盾；
- 推荐阅读路径；
- 高潜力故事种子；
- 最近更新；
- 授权方式。

操作：

- Star；
- Fork；
- 查看版本；
- 查看档案；
- 查看故事种子；
- 举报；
- 分享链接。

体验要求：

- 公开页应明显区别于工作台，偏浏览和判断价值；
- 但仍要保留仓库感，不做成小说展示页；
- 授权和 Fork 规则必须清楚可见。

### 8.8 Explore 页面

目标：帮助用户发现公开世界。

筛选维度：

- 类型；
- 风格；
- 标签；
- 最近更新；
- Star 数；
- Fork 数；
- 故事种子数量；
- 世界成熟度。

排序：

- 最新发布；
- 最近更新；
- 最多 Star；
- 最多 Fork；
- Trending，P1 可 Mock。

世界列表卡应展示：

- 世界名；
- 作者；
- 一句话设定；
- 标签；
- Star / Fork；
- 故事种子数量；
- 最近更新；
- 授权标识。

### 8.9 发布 / Push 页面

目标：让用户明确知道将公开什么，以及不会公开什么。

流程：

1. 选择创建新世界仓库或更新已有仓库；
2. 选择公开范围；
3. 选择发布内容；
4. 查看差异预览；
5. 填写更新说明；
6. 选择授权；
7. 确认发布；
8. 跳转公开世界页。

默认发布内容：

- 世界总览；
- 已确认世界规则；
- 已确认势力；
- 已确认角色；
- 已确认冲突；
- 已确认故事种子；
- README；
- 标签；
- 授权设置。

默认不发布：

- 原始对话记录；
- 本地草稿；
- 未确认设定；
- 私密备注；
- 模型配置；
- API Key；
- 本地日志；
- token 记录。

差异预览 P0：

- 新增几项设定；
- 修改几项设定；
- 删除几项设定；
- 新增几个故事种子。

体验要求：

- 用户必须能清楚区分“公开快照”和“完整同步”；
- 隐私提示不能隐藏在小字里；
- 重要风险操作需要二次确认。

### 8.10 设置与用量页面

Cloud 用量页显示：

- 当前余额；
- 本月消耗；
- 最近用量记录；
- 单次 Agent Run 消耗；
- 余额不足提示；
- 充值入口占位。

Local 模型设置页显示：

- MODEL_PROVIDER；
- MODEL_BASE_URL；
- MODEL_API_KEY，默认隐藏；
- MODEL_NAME；
- MAX_TOKENS；
- TEMPERATURE；
- CONTEXT_LIMIT；
- 模型连接测试结果。

Local 社区连接页显示：

- Access Token 输入；
- 当前连接账号；
- 连接测试；
- Token 保存状态；
- 断开连接；
- Push 权限状态。

## 9. Agent Mock 体验要求

### 9.1 Mock Agent 行为

Mock Agent 需要模拟真实调用过程：

- 创建 Agent Run；
- 显示正在分析上下文；
- 显示工具调用状态；
- 分段流式输出；
- 生成归档建议；
- 生成一致性提醒；
- 生成故事种子建议；
- 更新 usage；
- 完成或失败。

### 9.2 Mock 事件类型

前端应围绕以下事件心智设计：

- `agent.run.started`；
- `agent.text.delta`；
- `agent.tool.started`；
- `agent.tool.finished`；
- `agent.suggestion.created`；
- `agent.warning.created`；
- `agent.usage.updated`；
- `agent.run.completed`；
- `agent.run.failed`。

### 9.3 Mock 内容质量

Mock 文案必须能体现界坞的产品差异。

例如用户输入“一个世界里，记忆可以被买卖”后，Mock 应能生成：

- 至少 3 条世界规则；
- 至少 1 个一致性风险；
- 至少 3 个故事种子；
- 可保存的设定卡；
- 本轮上下文引用。

Mock 内容不能只有占位文字或无意义 Lorem Ipsum。

## 10. 关键交互状态

所有核心页面都需要考虑以下状态：

- 空状态；
- 加载中；
- 正在流式生成；
- 工具调用中；
- 保存成功；
- 保存失败；
- 网络错误；
- 模型不可用；
- 余额不足；
- 权限不足；
- 本地未连接社区；
- 有未保存建议；
- 有未发布更改；
- 发布成功；
- Fork 成功；
- 举报已提交。

状态反馈要求：

- 操作后必须有即时反馈；
- 长任务必须显示进度或阶段；
- 错误提示需要说明下一步可做什么；
- 危险操作需要确认；
- 不应让用户猜测当前数据是否已保存。

### 12.1 桌面端

桌面端是核心体验，应完整支持：

- 对话优先工作台；
- 可收起资产入口；
- 按需展开的辅助抽屉；
- 长文本编辑；
- 多列表浏览；
- 公开世界仓库阅读；
- 发布差异预览。

### 12.2 平板端

平板端可以压缩为：

- 单列或弱双列对话体验；
- 资产入口进入顶部菜单或左侧抽屉；
- 建议、上下文和编辑表单使用右侧或底部抽屉；

移动端不应把“资产 / 协作 / 沉淀”做成三块并列心智，而应保持“先对话，再处理建议，再进入资产页”的顺序。

## 14. 前端验收标准

### 14.1 创作闭环验收

给定用户输入：

“一个世界里，记忆可以被买卖。”

前端原型应能完成：

1. 创建世界；
2. 展示世界雏形确认卡；
3. 进入世界工作台；
4. Mock Agent 流式推演；
5. 展示至少 3 条可保存设定；
6. 展示至少 1 条一致性提醒；
7. 展示至少 3 个故事种子；
8. 用户保存设定到世界档案；
9. 用户保存故事种子到故事种子池；
10. 保存后的档案和种子页面能看到对应内容。

### 14.2 社区闭环验收

前端原型应能完成：

1. 浏览 Explore；
2. 打开公开世界仓库页；
3. 查看 Overview；
4. 查看公开档案；
5. 查看故事种子；
6. Star 世界；
7. Fork 世界；
8. Fork 后生成自己的私有世界状态；
9. 查看 Releases；
10. 提交举报。

### 14.3 发布 / Push 闭环验收

前端原型应能完成：

1. 从工作台进入发布或 Push；
2. 选择发布内容；
3. 明确展示不会公开的内容；
4. 展示实体级差异预览；
5. 填写更新说明；
6. 选择授权；
7. 确认发布；
8. 跳转公开世界仓库页；
9. 世界状态变为已公开或已 Push。

### 14.4 Local / Cloud 差异验收

前端原型应能表达：

- Cloud 显示余额、创作点、本次消耗；
- Local 显示模型连接状态；
- Local 显示社区 Access Token 连接；
- Local Push 明确是公开快照；
- Cloud Publish 是云端世界发布；
- 两者核心工作台体验保持一致。

## 15. 体验风险

### 15.1 跑偏成聊天工具

风险：用户只看到中间聊天框，不理解世界档案和故事种子的价值。

前端应对：

- Agent 输出后在消息内立即出现可操作建议；
- 对话顶部或消息尾部持续显示待处理建议数量；
- 保存后的资产在入口数量、Toast、档案页和种子页中可见；
- 首次保存时给出轻量引导，让用户理解“对话会沉淀为世界资产”；
- 保持档案、种子、冲突等独立页面的存在，避免产品退化为普通聊天记录。

### 15.2 跑偏成小说生成器

风险：用户以为产品用于直接写正文。

前端应对：

- 主操作围绕推演、收束、保存、生成种子；
- 不将“写正文”作为 P0 主路径；
- 故事种子页不做章节编辑。

### 15.3 Mock 太假

风险：前端看起来漂亮，但无法验证真实产品体验。

前端应对：

- Mock 数据必须有状态变化；
- Agent 输出必须流式；
- 保存、丢弃、Fork、发布必须影响页面数据；
- 至少覆盖错误和空状态。

### 15.4 信息过载

风险：世界工作台三栏内容过多，新用户不知道从哪里开始。

前端应对：

- 工作台默认采用对话优先布局；
- 首次进入时只给出轻量引导；
- 当前推荐操作明确；
- 左右辅助信息默认收起；
- 复杂信息跳转到独立页面或按需抽屉；
- 重要操作视觉层级突出；
- 复杂信息渐进展开。

## 16. 文案要求

文案应保持专业、清晰、有创作感。

推荐语气：

- “继续推演这个设定”
- “保存到世界档案”
- “发现 1 个潜在冲突”
- “从这个冲突生成故事种子”
- “本轮引用了 4 项世界资产”
- “以下内容将公开到界仓社区”

避免语气：

## 18. 总结

界坞前端第一阶段的目标，是做出一个能让人真实感受到产品成立的体验雏形。

这个雏形必须证明：

- 用户可以从一个模糊灵感开始；
- Agent 的价值体现在推演、挑刺和收束；
- 世界档案能成为长期资产；
- 故事种子来自世界内部矛盾；
- 社区不是作品发布站，而是世界仓库；
- Local 和 Cloud 是同一套创作体验下的不同责任边界。

只要前端能让用户走完这些路径，后端需求就会自然变得清晰。
