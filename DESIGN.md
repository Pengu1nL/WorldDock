# Design: WorldDock Story-First Pivot

Generated by /office-hours on 2026-06-22
Branch: main
Repo: WorldDock
Status: APPROVED
Mode: Startup (pre-product)

## Problem Statement

WorldDock 当前是"agent 辅助推演世界"的工具——用户创建世界，agent 帮助探索和扩展设定。但实际使用发现：**离开故事，世界设定是空洞的。** 一堆孤立的人物卡、地点描述、冲突大纲，缺少叙事生命力。

Pivot 方向：**让 agent 辅助故事推演，并从故事推演中沉淀世界资产。** 故事是交互的第一界面，世界资产是推演的副产品。

最终愿景：故事 → 世界资产 → AI 电影。从文字到视觉的完整创作 pipeline。

## Demand Evidence

- **创始人是用户本人**：跨形态创作者（网文 + AI 视觉叙事），当前用碎片化 AI 工具拼凑工作流
- **真实痛点**：写作过程中设定一致性全靠人肉检查，故事越长越失控
- **差异化**：现有工具要么是纯写作软件（Scrivener），要么是纯世界管理（WorldAnvil），没有把两者用 AI agent 串联起来的
- **⚠️ 尚未外部验证**：没有其他用户表达过付费或使用意愿。下一步需要验证

## Status Quo

当前方案：AI 工具（ChatGPT 辅助构思）+ 文档管理（Notion/Obsidian 管理设定）。信息割裂，来回切换。没有统一系统把故事推演和世界沉淀串起来。

## Target User & Narrowest Wedge

**目标用户**：跨形态创作者——需要同时管理长篇叙事和复杂世界观的人。第一阶段聚焦：
- 网文/长篇小说作者（日更/周更，设定庞杂）
- 后续延伸：TRPG DM、AI 视觉叙事者

**Narrowest Wedge**：将 WorldDock 重构为叙事优先架构。Narrative（故事）提升为核心实体——用户创建的不是"世界设定集"，而是"故事"。World 降级为故事的容器（一个世界观可以容纳多个故事）。Asset 不再是独立管理的实体，而是从叙事推进中自动派生、持续演化的投影。用户在写作中自然沉淀世界。

## Constraints

- 复用现有技术栈（NestJS API + React Web + Prisma + Agent 系统）
- Narrative 是核心实体，World 是容器，Asset 是叙事状态的派生投影
- Agent 保持"推演引擎"定位——参谋，不是代笔
- 一个 World 可以有多个 Narrative（1:N）
- 允许对现有 World/Asset 模块做结构性改造，不做表面修补

## Key Definitions

**推演 (Progression)** — Agent 读取最新章节后执行三个动作：
1. **状态更新**：追踪每个角色的当前状态（存活/死亡/位置/关系变化），生成角色状态快照
2. **一致性标记**：对比章节内容与现有世界资产，标记矛盾（如"第3章说角色A在京城，第7章说他在江南，中间没有旅行描写"）
3. **设定缺口发现**：识别文中提到但尚未建档的设定（如"主角使用了'灵脉'这个概念，但世界资产中没有灵脉的定义"）

产出格式为结构化 JSON，用户可在 UI 中逐条确认/拒绝/修改。


## Premises (all confirmed)

1. **故事先行，世界从故事中生长** — 创作体验是"写"不是"管"
2. **Agent 是推演引擎，不是内容生成器** — 推演后果、检查一致性、发现设定缺口。用户保留创作主权
3. **资产结构化且 AI 可消费** — 包含视觉化属性，服务最终的 AI 电影目标

## Approaches Considered

### Approach A: 故事层适配 (Minimal Viable)

在现有架构上加 Story 模块。Story 是新的一级实体，agent session 挂载到故事上，资产从故事中由 agent 自动提取。

Effort: S-M | Risk: Low

Pros:
- 改动最小，复用现有 World/Asset/Agent 基础设施
- 快速验证"故事驱动"假设，1-2 周出可用的原型
- 如果方向不对，回滚成本低

Cons:
- 架构债务——Story 挂在 World 下，Asset 仍然是独立实体
- 从故事到资产的推导链路不够流畅，资产手动关联只是折中方案
- 后续如果要真正实现"资产自动派生"，需要二次重构

### Approach B: 叙事优先重构 (Ideal Architecture — SELECTED)

将 Narrative 提升为核心实体，World 降级为容器，Asset 成为叙事状态的投影。

Effort: L | Risk: Med

Pros:
- 数据模型干净，"故事是源头"的理念贯彻到底
- Asset 是自动派生的投影，不是独立管理的实体——用户写作即沉淀世界
- 架构一步到位，无需后续二次重构

Cons:
- 对现有 World/Asset 模块做结构性改造，开发周期 2-3 周
- 需要迁移现有数据和 UI 交互范式

### Approach C: 时间线事件引擎 (Creative/Lateral)

可视化时间线交互，事件碰撞涌现故事。更像游戏引擎。

Effort: XL | Risk: High

Pros:
- 交互范式创新，差异化极强
- 天然适合"agent 推演后果"的玩法

Cons:
- 开发成本最高，且用户学习曲线陡峭
- 放弃了"文字创作"这个最熟悉的交互范式

## Recommended Approach

**Approach B — 叙事优先重构。** Narrative 是第一公民。World 是轻量容器。Asset 是叙事状态的派生投影。

### 核心理念

当前架构：World → Asset（setting/seed/conflict），Agent 围绕 World 工作。

新架构：**Narrative → Chapter → Agent 推演 → Asset 投影**

用户交互的主界面不再是"世界管理后台"，而是"故事写作界面"。用户写章节 → Agent 推演后果 → 角色/地点/事件等资产从叙事中自然析出。用户不需要手动创建 Asset——写得越多，世界越丰富。

### Data Model

```prisma
// ============ 核心实体：叙事 ============

model Narrative {
  id        String      @id @default(uuid())
  worldId   String?
  world     World?      @relation(fields: [worldId], references: [id])
  title     String
  synopsis  String?
  status    NarrativeStatus @default(draft)
  createdAt DateTime    @default(now())
  updatedAt DateTime    @updatedAt

  chapters  Chapter[]
  assets    NarrativeAsset[]

  @@index([worldId])
}

enum NarrativeStatus {
  draft
  in_progress
  completed
  archived
}

model Chapter {
  id          String     @id @default(uuid())
  narrativeId String
  narrative   Narrative  @relation(fields: [narrativeId], references: [id])
  order       Int        // 章节序号
  title       String
  content     String     // 正文（Markdown）
  wordCount   Int        @default(0)
  status      ChapterStatus @default(draft)
  createdAt   DateTime   @default(now())
  updatedAt   DateTime   @updatedAt

  progressionRuns ProgressionRun[]

  @@unique([narrativeId, order])
  @@index([narrativeId])
}

enum ChapterStatus {
  draft
  completed
  revised
}

// ============ 派生实体：叙事资产 ============

model NarrativeAsset {
  id          String   @id @default(uuid())
  narrativeId String
  narrative   Narrative @relation(fields: [narrativeId], references: [id])
  kind        AssetKind
  name        String   // 资产名称（角色名/地点名/事件名）
  summary     String   // 当前已知的摘要
  body        String?  // 详细信息（Markdown）
  tags        String[] @default([])
  // 视觉化属性（渐进增强，初始可选）
  appearance  String?  // 外貌/外观描述
  mood        String?  // 氛围/气质关键词
  createdAt   DateTime @default(now())
  updatedAt   DateTime @updatedAt

  versions    AssetVersion[]
  // 资产之间的关联
  relationsFrom NarrativeAssetRelation[] @relation("source")
  relationsTo   NarrativeAssetRelation[] @relation("target")

  @@unique([narrativeId, kind, name])
  @@index([narrativeId])
}

enum AssetKind {
  character   // 角色
  location    // 地点
  item        // 道具/物品
  event       // 事件
  concept     // 概念/设定（功法、魔法体系、科技等）
  faction     // 组织/势力
}

// 资产版本历史：记录每章推演后资产的变化
model AssetVersion {
  id        String         @id @default(uuid())
  assetId   String
  asset     NarrativeAsset @relation(fields: [assetId], references: [id])
  chapterId String         // 哪一章的推演产生了这个版本
  chapter   Chapter        @relation(fields: [chapterId], references: [id])
  snapshot  String         // JSON：该版本时的完整资产状态
  diff      String?        // JSON：相对上一版本的变更
  createdAt DateTime       @default(now())
}

// 资产关联
model NarrativeAssetRelation {
  id            String         @id @default(uuid())
  sourceAssetId String
  source        NarrativeAsset @relation("source", fields: [sourceAssetId], references: [id])
  targetAssetId String
  target        NarrativeAsset @relation("target", fields: [targetAssetId], references: [id])
  relationType  String         // "ally", "enemy", "located_in", "owns", "member_of", "related_to"
  description   String?        // 关系描述
  createdAt     DateTime       @default(now())
}

// ============ 推演记录 ============

model ProgressionRun {
  id           String   @id @default(uuid())
  chapterId    String
  chapter      Chapter  @relation(fields: [chapterId], references: [id])
  inputContext String   // JSON：推演时的输入上下文（章节全文摘要 + 已知资产快照）
  outputRaw    String   // JSON：Agent 原始输出
  status       ProgressionRunStatus @default(pending_review)
  createdAt    DateTime @default(now())

  @@index([chapterId])
}

enum ProgressionRunStatus {
  pending_review  // 等待用户审核
  confirmed       // 用户确认，资产已更新
  rejected        // 用户拒绝
  partially_confirmed // 部分确认
}
```

### World 实体的新角色

World 保留但简化为轻量容器：

```prisma
model World {
  id        String   @id @default(uuid())
  name      String
  type      String   // "novel", "ttrpg", "screenplay", etc.
  summary   String
  tags      String[] @default([])
  maturity  Int      @default(0)
  createdAt DateTime @default(now())
  updatedAt DateTime @updatedAt

  narratives Narrative[]
}
```

现有的 `ArchiveEntry`、`StorySeed`、`Conflict` 表保留但不再作为核心交互路径。它们成为"官方资产"的存档——用户可以从叙事资产中 select 某些提升为官方设定。

### Agent 推演 Pipeline（不变的核心流程，适配新数据模型）

1. **构建 Context**：当前章节全文 + 上一个 ProgressionRun 的角色状态快照 + 已有 NarrativeAsset 摘要表
2. **Agent 推演**（单次 LLM 调用）：输出结构化 ProgressionOutput JSON
3. **生成建议**：后端解析 JSON → 映射为 Asset create/update/version 操作
4. **用户审核**：所有建议以卡片流展示，逐条确认/拒绝/修改
5. **写入**：确认后创建 AssetVersion，更新 NarrativeAsset 当前状态

**Agent 输出 Schema（与 A 方案相同，适配新 AssetKind）：**

```typescript
type ProgressionOutput = {
  assetChanges: {
    kind: "character" | "location" | "item" | "event" | "concept" | "faction";
    name: string;
    operation: "create" | "update";
    summary: string;              // 当前已知
    body?: string;                // 详细信息
    appearance?: string;          // 外貌描述（character/location）
    mood?: string;                // 氛围（location/event）
    tags?: string[];
    relationChanges?: {
      targetName: string;
      relationType: string;
      description: string;
    }[];
  }[];
  consistencyFlags: {
    severity: "warning" | "error";
    claim: string;
    conflictWith: string;
    suggestion: string;
  }[];
  narrativeObservations: {
    // Agent 对故事走向的观察，不直接修改资产
    observation: string;
    implication: string;  // "这意味着..."
    suggestedAction?: string;
  }[];
};
```

### API 设计

```
// Narratives (新主实体)
POST   /api/narratives                        — 创建叙事
GET    /api/narratives                        — 列出叙事（可按 world 过滤）
GET    /api/narratives/:id                    — 获取叙事详情（含资产摘要）
PATCH  /api/narratives/:id                    — 更新叙事
DELETE /api/narratives/:id                    — 删除叙事

// Chapters
POST   /api/narratives/:nid/chapters           — 创建章节
PATCH  /api/narratives/:nid/chapters/:cid      — 更新章节
DELETE /api/narratives/:nid/chapters/:cid      — 删除章节
POST   /api/narratives/:nid/chapters/:cid/progress — 触发推演

// Assets (只读为主，少量手动修正)
GET    /api/narratives/:nid/assets             — 列出叙事资产
GET    /api/narratives/:nid/assets/:aid        — 资产详情 + 版本历史
PATCH  /api/narratives/:nid/assets/:aid        — 手动修正资产（可选）
GET    /api/narratives/:nid/assets/:aid/versions — 版本历史

// Progression
GET    /api/narratives/:nid/progressions       — 推演历史
GET    /api/narratives/:nid/progressions/:pid  — 单次推演详情
POST   /api/narratives/:nid/progressions/:pid/confirm   — 确认全部
POST   /api/narratives/:nid/progressions/:pid/confirm-item/:itemId — 逐条确认
POST   /api/narratives/:nid/progressions/:pid/reject    — 拒绝

// Worlds (保留，降级为容器)
GET    /api/worlds                             — 列出世界
POST   /api/worlds                             — 创建世界
GET    /api/worlds/:id                         — 世界详情（含叙事列表）
```

### Frontend 结构

新路由结构：
- `/:narrativeId` — 故事写作页（主要交互界面）
- `/worlds/:worldId` — 世界概览（叙事列表 + 资产总览）
- `/worlds` — 世界列表

**故事写作页**（核心页面）：
- 左侧：章节导航（可拖拽排序、新建、删除）
- 中间：Markdown 编辑器（正文写作区）
- 右侧：推演面板（Progression 结果卡片流）
  - 角色更新卡片（可展开查看版本 diff）
  - 一致性警告卡片（红色/黄色，可忽略）
  - 叙事观察卡片（Agent 对故事走向的建议）
  - 每条可单独确认/拒绝

**世界概览页**：
- 叙事时间线（该世界下所有故事的章节推进情况）
- 资产关系图（角色、地点、势力之间的关系网络，D3/vis.js）

### 迁移策略

现有数据保留不删：

1. 现有 `World` → 保留，新增 `Narrative` 1:N 关联
2. 现有 `ArchiveEntry`/`StorySeed`/`Conflict` → 保留为"官方资产存档"，新增 `NarrativeAsset` 表
3. 用户可以将 NarrativeAsset 提升为官方资产（promote to official）
4. 现有 Agent Session 的 `world_exploration` kind 保留，新增 `story_progression`

|------|------|

## Open Questions

- **Agent 推演触发时机**：MVP 阶段手动触发（"Run Progression"按钮，完成一章后点击）。后续考虑：章末自动触发，或定时触发"世界时钟推进"
- **提取粒度**：MVP 阶段 agent 全量扫描每章结尾，但所有产出需用户确认。后续：用户可以标记特定段落让 agent 关注
- ~~故事和世界的关系~~ → 已决定：1:N（一个 World 可以有多个 Story）

## Success Criteria

- 用户能在 WorldDock 中写完一个完整的故事章节
- Agent 能从章节内容中自动提取至少一个世界资产建议
- 所有 agent 建议需用户确认后才写入（数据安全底线）
- 提取的资产自动关联到现有一致性检查系统

## Data Safety

- Agent 的所有资产修改以"建议"形式呈现，不直接写入
- 每次推演生成一个 `ProgressionRun` 记录，包含输入上下文和完整输出，可回溯
- 用户可一键拒绝单条建议或回滚整轮推演
- 资产被 agent 修改后，旧值保留在 `WorldAssetPatch` 中（复用现有 official assets patch 机制）

## Deferred (post-MVP)

- 角色/地点/事件的视觉属性（外貌描述 → 图生成 prompt）
- 跨章节追踪（角色弧光、伏笔回收）
- 导入/导出（与现有 World Package 格式整合）
- 离线支持

## Distribution Plan

现有 Web 应用 + CLI 工具。Web 端是主要交付渠道。不需要新增分发机制。

## Dependencies

- 无外部阻塞
- 依赖现有 Prisma schema 迁移流程
- 需要更新 Agent 的 system prompt 以支持"故事推演"模式

## What I noticed about how you think

- "离开故事的设定会非常空洞"——这不是一个功能需求，这是一个产品哲学。你在用户访谈里都不一定能听到这么精准的表述
- 你的最终愿景不是"做个写作工具"而是 AI 电影——故事→资产→视觉的完整 pipeline。这个 vision 比你当前的 MVP 大得多
- 你一开始选了 Approach A（最小改动），看了完整设计后又决定切到 B（叙事优先重构）。这不是犹豫——这是理解了 tradeoff 之后做了有意识的架构决策。你知道 B 的成本更高，但你选择为长期架构质量付出这个代价
- 当前最大的风险不是技术，是需求验证。你还没有和"不是你自己"的用户聊过

### Architecture Changes from Review

**原设计 → 修订后：**

1. **Agent 调用模式**：单次 LLM 调用 → 多轮 tool-use 会话（渐进式披露）。Agent 有 `list_characters`、`get_asset`、`get_previous_chapter_snapshot` 工具
2. **ProgressionRun 模型**：移除独立 model → 合并入 AgentSession（kind: "story_progression"）
3. **NarrativeAsset unique 约束**：移除 `@@unique([narrativeId, kind, name])` → 写入时计算 name embedding，推演时做向量去重建议
4. **推演触发**：同步阻塞 → 异步（POST 创建 AgentSession → 返回 sessionId → 前端轮询结果）
5. **旧 Asset 表**：保留但不接入新 API，NarrativeAsset 是唯一资产系统
